This invention relates to cooperative multi-agent artificial intelligence substrate architectures; specifically, to methods and systems for memory expansion via substrate-routed topic delivery, hierarchical authority management via recursive ring-structure instantiation, social-graph-anchored decision governance, cooperative federation library cost-savings architectures, persistent-substrate agent management, and install-time substrate self-verification primitives.

Background of the Invention

The Memory Cost Explosion Problem

AI agent systems face a structural cost explosion as the corpus of canonical knowledge grows. Under conventional approaches, an agent must load ALL canonical knowledge into its context window at session start to be able to answer questions about any topic. As the knowledge base grows from dozens to hundreds to thousands of topics, context cost grows linearly with topic count — even though any single agent session addresses only a small subset of available topics.

This creates a false tradeoff: either (a) maintain a small knowledge base to control costs (limiting capability), or (b) pay exponentially growing session-start costs to maintain full-topic coverage (limiting commercial viability). Neither path is acceptable for a cooperative substrate serving members at $5/year.

The Memory Loading Bottleneck

Compounding the cost explosion, existing systems load canonical memory once at session start in a batch operation. When new canonical knowledge is added between sessions, each subsequent session-start becomes more expensive. Existing AI agent session-management systems do not distinguish between “topics relevant to this task” and “all topics ever known.” The entire canon is treated as equally load-bearing for every session, whether or not the topics addressed are relevant.

The Vendor Lock-In and Authority Fragmentation Problem

Multi-agent AI systems operating across multiple vendor platforms (Anthropic, OpenAI, Google, Perplexity) lack portable authority structures. Authority rules established in one vendor’s session environment cannot be portably enforced in another. Downstream authorities (project owners, sub-project coordinators) cannot safely extend a parent authority structure without risking override of non-overrideable parent rules.

Social-graph entities (guilds, tribes, families) that naturally cross vendor-session boundaries have no mechanism for recording decisions as first-class substrate artifacts. Decisions become ephemeral conversation artifacts rather than durable, queryable, provenance- anchored records.

The Cooperative Access and Cost-Sharing Problem

AI substrate costs — primarily vendor API spend and infrastructure — have historically been born by individual practitioners in isolation. No mechanism exists for cooperative cost-sharing that preserves each member’s sovereignty while enabling cross-member content reuse. Existing sharing mechanisms (SaaS subscriptions, open-source repositories) either extract commercial rents (subscription lock-in) or lack the governance discipline to keep shared content current (open-source drift).

The Installation Verification Problem

Users installing AI substrate platforms currently have no in-band mechanism to verify that the installation completed correctly and all components are functional. Existing installation verification is either absent (trust the installer) or requires separate manual testing by technically sophisticated users. For cooperative substrate platforms targeting non-expert users, this creates an installation failure silent mode — the substrate appears installed but operates in degraded state.

Detailed Description of the Invention

Section I — Substrate-Routed Memory Expansion (#2336)

Innovation Title: Substrate-Routed Memory Expansion: O(Relevant-Topics-Per-Task) Agent Memory Index with Wrasse-Routed Canonical Delivery

Classification: - Innovation Number: #2336 (candidate; verify against A&A formal ledger at conversion) - Crown Jewel Candidate: YES - Patent Provisional: Prov 16 - Date of Conception: BP005 (2026-03-20 through 2026-04-30 consolidation window) - Conceived by: Jonathan Jones (Founder & General Manager) - Reduction to Practice: KN042 Pod O BP005 (commit 0696f31)

The Architectural Problem (enabling disclosure):

Under conventional AI agent memory management, an agent’s canonical knowledge file (e.g., MEMORY.md) must contain entries for every topic the agent might need to address in any session. As canonical knowledge grows, the memory file grows proportionally. A 200-topic canon requires a 200-entry file; a 1,000-topic canon requires a 1,000-entry file. Context cost grows linearly with topic count regardless of task relevance.

The Substrate-Routed Solution:

Substrate-Routed Memory Expansion separates the memory INDEX from memory CONTENT:

MEMORY.md (index layer — stays small):
  - topic: "Federation Library L5 algorithm"
    trigger_class: vocabulary
    trigger_pattern: "Federation Library|L5|97\\.2%"
    content_ref: federation_library_l5
    (full content NOT loaded — wrasse delivers on trigger match)

  - topic: "Prov 16 filing details"
    trigger_class: vocabulary
    trigger_pattern: "Prov 16|provisional 16|KN071"
    content_ref: prov_16_details
    (full content NOT loaded)

The index entry is ≤5 lines. The full topic content (which may be 50-500 lines) lives in the Wrasse substrate. When a session’s incoming context matches a trigger pattern, the Wrasse Scribe (#2332, already in Prov 15) delivers the full topic content — and only that topic’s content — into the session context.

Measured empirical result (KN042 Pod O BP005, commit 0696f31): - MEMORY.md compacted: 405 lines → 141 lines (65% reduction) - Substrate canon simultaneously grew by 60+ Eblets (no ceiling hit) - Trigger test suite: 8/8 triggers fired correctly on test context injection - Full test suite: 33/33 tests passing at commit - Context cost profile: session pays for relevant-topics-only, not total-canon

The scaling invariant: As canon grows from 60 to 600 to 6,000 Eblets, MEMORY.md index size grows by one 5-line entry per new topic — not by the full topic content. Total index size approaches O(N × 5) lines, not O(N × full-topic-length). The ratio of index-size to total-canon-size decreases monotonically as canon grows.

Composition with Wrasse Pre-Injection (#2332, Prov 15): Wrasse is the delivery mechanism. Substrate-Routed Memory Expansion is the ARCHITECTURAL PATTERN — the decision to store canonical topic content in the substrate (not inline in the index) and route topic delivery through trigger-matched injection. Wrasse implements the trigger-matching and delivery mechanics; Substrate-Routed Memory Expansion is the system architecture that makes Wrasse’s delivery pattern the load-bearing memory model for the entire substrate.

Scope wedge — variation coverage (Pawn Q1 defense):

Variation 1: Index file is not MEMORY.md but any agent-configuration file serving the memory-index role (system prompt, initialization file, settings YAML, etc.)

Variation 2: Trigger matching is not regex-first but embedding-based, BM25-based, or hybrid — the routing mechanism is implementation-variable; the O(relevant-topics-per-task) scaling property is the architectural invariant.

Variation 3: Content delivery is not pre-injection but retrieval-on-demand — agent queries substrate for topic content mid-session rather than receiving pre-injected block. The index-content separation and O(relevant-topics) cost profile are preserved.

Variation 4: Index lives in-substrate (Pheromone/JSONL/vector DB) rather than in a human-readable flat file, with programmatic topic resolution at session-open.

Section II — Ring of Three Golden Eblets (#2337)

Innovation Title: Ring of Three Golden Eblets: Three-Component Top-of-Substrate Authority Structure with Non-Overrideable Marked Exception Enforcement

Classification: - Innovation Number: #2337 (candidate) - Crown Jewel Candidate: YES - Patent Provisional: Prov 16 - Conceived by: Jonathan Jones (Founder & General Manager) - Reduction to Practice: BP005 base ratification — 3 Golden Eblets at ~/.claude/state/eblets/CANON/GOLDEN/ (282 combined lines) + federation canon project_ring_of_three_golden_eblets_deck_card_medallion_federation_canon.md (450+ lines)

The Authority Architecture:

The Ring of Three Golden Eblets is a three-component authority structure forming the top layer of every substrate deployment:

GOLDEN EBLET 1 — Canon:
  Content: Inviolable principles. These are not adjustable by any downstream authority.
  Enforcement: Marked Exception — LB-source artifact QR ALWAYS routes to Canon.
  Override status: NON-OVERRIDEABLE by any downstream layer.

GOLDEN EBLET 2 — Platform Rules:
  Content: Platform-wide operational directives. Govern ALL agents on the platform.
  Enforcement: Applies uniformly across all projects on the platform.
  Override status: NON-OVERRIDEABLE at project layer (Part A of each platform rule).

GOLDEN EBLET 3 — Project Rules:
  Content: Project-specific operational directives within Canon + Platform Rule bounds.
  Enforcement: Applies within the project scope; cannot override Eblet 1 or 2.
  Override status: Adjustable only within the headroom left by Eblets 1 + 2.

The Marked Exception mechanism: When an agent receives an LB-source artifact (identified by QR code, Call Sign, or canonical provenance marker), the agent’s context-loading sequence MUST route to Canon (Eblet 1) regardless of any downstream rule modification. This non-overrideable routing is the Marked Exception: the project owner cannot configure their Project Rules to bypass Canon routing for LB-source artifacts. This is structurally enforced (not policy-enforced) — the Wrasse trigger-class for LB-source artifacts points unconditionally to Eblet 1.

Recursive instantiation: The Ring of Three is not a fixed three-layer structure. At each successive authority layer (L2 dual-entity, L3 project owner, L4 sub-project, L5 member), the authority layer instantiates its own Ring of Three Golden Eblets for its scope. The parent’s Canon (Eblet 1) propagates non-overrideable through every child Ring. The child adds its own Part A (non-overrideable within the child scope) and Part B (adjustable within the child scope). This recursive composition is the subject of #2338 Multi-Layer Authority Recursion.

Scope wedge — variation coverage: - Variation: Ring of Two (Canon + Unified Rules) — reduced composition with same enforcement - Variation: Ring of Four or N — extended composition; same Marked Exception mechanism applies - Variation: Non-file-based implementation — Ring expressed as database records, API policy objects, smart contract rules, or equivalent structured artifacts - Variation: Multi-substrate Ring composition — Ring instantiated across vendor-heterogeneous agent environments with cross-vendor Marked Exception enforcement

The compound rules-stack:

At each layer L_k, the effective rule set is the union: - {L1 Part A} (non-overrideable, always propagated) - {L2 Part A} (non-overrideable through all L2+ layers) - … (each parent layer’s Part A propagates downward) - {L_k Part A} (non-overrideable within L_k scope) - {L_k Part B} (adjustable within headroom of all parent Part A rules)

No Part B at layer L_k may modify or override any Part A from any parent layer. The mechanical enforcement at the Wrasse level: when an agent at layer L_k loads its Ring of Three, the Wrasse pre-injection sequence loads all parent Part A Eblets before loading L_k’s own Ring. Downstream agents cannot unload the parent Part A Eblets.

Layer Addressing Scheme (KN045): Each layer has a canonical address:

L1.{substrate_root}
L2.{entity_name}.{L1}
L3.{project_name}.{L2}.{L1}
L4.{subproject_name}.{L3}.{L2}.{L1}

A method for decoupling AI agent memory-index cost from total canonical topic count, comprising: - (a) maintaining a memory index of N topics, wherein each entry comprises a topic identifier and a trigger pattern, and wherein full topic content is stored externally from the index; - (b) at each AI agent session start, loading only the memory index, not full topic content; - (c) detecting trigger-pattern matches in the agent’s incoming session context against the memory index using pattern-matching; - (d) delivering full topic content for each matched trigger, and only for matched triggers, into the agent’s context before the agent’s first reasoning token fires; - (e) whereby the agent’s context cost at session start is O(relevant-topics-per-session) rather than O(total-topics), and the memory index size remains bounded independently of total canonical topic count.

Claim 2 (Substrate-Routed Memory Expansion — Dependent: Scalability Property)

The method of Claim 1, wherein as total canonical topic count grows from N to M (M > N), the memory index size grows by at most K × (M - N) characters, where K is a constant bounded by the mean trigger-pattern length, independent of the full topic content of the added topics.

Claim 3 (Substrate-Routed Memory Expansion — System Claim)

A system for substrate-routed AI agent memory expansion, comprising: - a memory index store containing per-topic entries with trigger patterns and external-content references, structured to permit O(relevant-topics-per-session) index loading; - a trigger-matching engine that evaluates incoming agent context against stored trigger patterns and identifies matched topics; - a content delivery module that retrieves and delivers full topic content exclusively for matched topics before agent reasoning begins; - an empirical validation harness confirming that index-size growth is decoupled from total-topic-count growth across at least two measured corpus-expansion events.

Claim 4 (Ring of Three Golden Eblets — Independent Method Claim)

A method for hierarchical authority management in a multi-agent AI substrate platform, comprising: - (a) defining a three-component top-of-substrate authority structure consisting of a first component encoding inviolable principles, a second component encoding platform-wide operational directives, and a third component encoding project-specific operational directives; - (b) enforcing that the first component is non-overrideable by any downstream authority layer or agent configuration; - (c) enforcing that the second component is non-overrideable at the project layer; - (d) for any agent-source artifact identified by a canonical provenance marker, routing agent context-loading unconditionally to the first component before any other context loads; and - (e) permitting downstream authority layers to define their own third-component rules only within the headroom left by the first and second components.

Claim 5 (Ring of Three Golden Eblets — Dependent: Recursive Instantiation)

The method of Claim 4, wherein each successive authority layer in a multi-layer hierarchy of at least two layers instantiates its own three-component authority structure per Claim 4, and wherein the first component of every child layer inherits and preserves the non-overrideable rules of the first component of every parent layer in the hierarchy.

Claim 6 (Multi-Layer Authority Recursion — Independent Method Claim)

A method for substrate-as-a-service deployment with compound authority-rule inheritance, comprising: - (a) defining at least two authority layers L_k and L_{k+1} in a hierarchical substrate deployment, each authority layer comprising a three-component authority structure per Claim 4; - (b) enforcing that the non-overrideable rules of L_k propagate to L_{k+1} without modification; - (c) assigning a canonical layer address to each authority layer, the address comprising the layer identifier and all parent layer identifiers in ordered sequence; - (d) validating, before any authority directive from a lower layer takes effect, that no circular authority reference exists in the layer graph that would grant a child layer authority over a parent layer; and - (e) enabling each authority layer to define its own brand identifiers, stamps, QR codes, and IP ledger entries, distinct from those of parent layers.

Claim 8 (Social-Authority DAG + Pheromone-Anchored Decisions — Independent Method Claim)

Claim 16 (Augur Living Gate — Independent Method Claim)

A method for substrate-event-driven freshness gating in an AI memory substrate, comprising: - (a) recording, in a stigmergic substrate index, the timestamp of each write operation to any canonical substrate record associated with a topic; - (b) recording, for each agent consultation of a topic, the timestamp of that consultation; - (c) when an agent requests a topic’s canonical content, retrieving the topic’s most recent write timestamp from the substrate index; - (d) if the most recent write timestamp postdates the most recent consultation timestamp, re-deriving the topic’s canonical content before returning it; and - (e) if the most recent write timestamp predates the most recent consultation timestamp, returning the prior derivation without re-derivation; - (f) whereby re-derivation is triggered by actual substrate state change, not by elapsed clock time.
