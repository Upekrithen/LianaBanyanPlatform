## Background of the Invention

### The Memory Cost Explosion Problem

AI agent systems face a structural cost explosion as the corpus of canonical knowledge grows.
Under conventional approaches, an agent must load ALL canonical knowledge into its context
window at session start to be able to answer questions about any topic. As the knowledge base
grows from dozens to hundreds to thousands of topics, context cost grows linearly with
topic count — even though any single agent session addresses only a small subset of available
topics.

This creates a false tradeoff: either (a) maintain a small knowledge base to control costs
(limiting capability), or (b) pay exponentially growing session-start costs to maintain
full-topic coverage (limiting commercial viability). Neither path is acceptable for a
cooperative substrate serving members at $5/year.

### The Memory Loading Bottleneck

Compounding the cost explosion, existing systems load canonical memory once at session start
in a batch operation. When new canonical knowledge is added between sessions, each subsequent
session-start becomes more expensive. Existing AI agent session-management systems do not
distinguish between "topics relevant to this task" and "all topics ever known." The entire
canon is treated as equally load-bearing for every session, whether or not the topics addressed
are relevant.

### The Vendor Lock-In and Authority Fragmentation Problem

Multi-agent AI systems operating across multiple vendor platforms (Anthropic, OpenAI, Google,
Perplexity) lack portable authority structures. Authority rules established in one vendor's
session environment cannot be portably enforced in another. Downstream authorities (project
owners, sub-project coordinators) cannot safely extend a parent authority structure without
risking override of non-overrideable parent rules.

Social-graph entities (guilds, tribes, families) that naturally cross vendor-session
boundaries have no mechanism for recording decisions as first-class substrate artifacts.
Decisions become ephemeral conversation artifacts rather than durable, queryable, provenance-
anchored records.

### The Cooperative Access and Cost-Sharing Problem

AI substrate costs — primarily vendor API spend and infrastructure — have historically been
born by individual practitioners in isolation. No mechanism exists for cooperative cost-sharing
that preserves each member's sovereignty while enabling cross-member content reuse. Existing
sharing mechanisms (SaaS subscriptions, open-source repositories) either extract commercial
rents (subscription lock-in) or lack the governance discipline to keep shared content
current (open-source drift).

### The Installation Verification Problem

Users installing AI substrate platforms currently have no in-band mechanism to verify that
the installation completed correctly and all components are functional. Existing installation
verification is either absent (trust the installer) or requires separate manual testing by
technically sophisticated users. For cooperative substrate platforms targeting non-expert
users, this creates an installation failure silent mode — the substrate appears installed but
operates in degraded state.

## Summary of the Invention

This disclosure presents an integrated cooperative AI substrate architecture organized under
the **Cathedral Adoption Pathway (CAP)** strategic frame: AI practitioners rationally migrate
toward cooperative substrate architectures because the Cathedral provides higher reliability,
lower cost, and cooperative governance benefits that proprietary alternatives cannot match.
The inventions in this filing are the CAP-implementing primitives.

The eight primary architectural ratifications disclosed herein compose as a single system:

**1. Substrate-Routed Memory Expansion (#2336):** Decouples agent memory index cost from
total topic count by delivering content only when a topic surfaces in the current task context.
Memory index stays small (≤200 lines) while total canonical knowledge grows without ceiling.
Context cost is paid only for topics relevant to the current task — O(relevant-topics-per-task)
instead of O(total-topics).

**2. Ring of Three Golden Eblets (#2337):** A top-of-substrate authority structure consisting
of exactly three components — Canon (inviolable principles), Platform Rules (platform-wide
operational directives), and Project Rules (project-specific operational directives) — with
a Marked Exception enforcement layer that routes LB-source artifacts unconditionally to Canon
regardless of downstream override attempts.

**3. Multi-Layer Authority Recursion (#2338):** A recursive instantiation model in which
each successive authority layer deploys its own Ring of Three Golden Eblets with its own
brand, stamps, and QR codes anchored to its own IP Ledger. The rules-stack compounds
non-overrideable parent rules with each successive layer's own Part A (non-overrideable)
and Part B (adjustable within parent-Part-A bounds).

**4. Social-Authority DAG + Pheromone-Anchored Decision Governance (#2339):** A method for
cross-cutting social-authority overlays (Guild, Tribe, Family, Helm) on the recursive
substrate-layer hierarchy. Each social unit maintains sovereignty over decision-mechanism
(HOW decisions are reached) and decision-outcome (WHOM the decision addresses). Decisions
are recorded as first-class Pheromone substrate records anchored to a Golden Tablet — not
as mutations of existing Tablet records.

**5. AGPL Cathedral Free / Federation Library Cooperative Architecture (#2340):** A dual-value
proposition architecture in which (a) the base substrate is AGPL v3 free, full-featured, and
ungated, while (b) cross-member cooperative library access is opt-in via cooperative membership
at $5/year identical for all members. Network effect compounds as member count grows; a
5-layer cost-savings algorithm (Federation Library B127 algorithm) delivers measured 97.2%
cost reduction at N≥2 members with theoretical ceiling approaching 98.3% at scale.

**6. LB Frame Broadcast Funnel (#2341):** A 5-stage substrate adoption funnel — Install (free,
one-click), Demo (substrate-pre-loaded empirical receipt), User-data Test (visitor's own data),
Join CTA ($5/year cooperative membership), Send-to-someone CTA (member-issued viral propagation
under Sponsorship Mark canon) — in which each stage is substrate-verifiable.

**7. Knight-Has-No-Shadows / Bishop-Only Persistent-Substrate Operations (#2342):** A formal
distinction between the persistent-substrate AI agent class (with hook architecture enabling
spawn-on-SessionStart / kill-on-SessionEnd persistent daemons = "Shadows") and the per-session
AI agent class (without hook architecture). LIGHTHOUSE 8/2 instrumentation — 8 production
cylinders + 2 instrumentation Scribes — requires the persistent-substrate agent class.

**8. Walkaround Universal Install-Time Preflight Primitive (#2347):** A universal substrate
self-verification primitive that fires at install-time, verifies all substrate components
against canonical integrity criteria, and emits a tabular empirical receipt (PASS/WARN/FAIL
per component) as the installation completion demonstration. The empirical receipt replaces
marketing claims; the substrate proves its own integrity in ≤5 seconds.

These eight compose with four BP004 substrate primitives (#2343–#2346) and four B134-era
innovations (#2287–#2289, #2348–#2350) to form the complete cooperative AI substrate
described in the detailed description below.

## Detailed Description of the Invention

### Section I — Substrate-Routed Memory Expansion (#2336)

**Innovation Title:** Substrate-Routed Memory Expansion: O(Relevant-Topics-Per-Task)
Agent Memory Index with Wrasse-Routed Canonical Delivery

**Classification:**
- Innovation Number: #2336 (candidate; verify against A&A formal ledger at conversion)
- Crown Jewel Candidate: YES
- Patent Provisional: Prov 16
- Date of Conception: BP005 (2026-03-20 through 2026-04-30 consolidation window)
- Conceived by: Jonathan Jones (Founder & General Manager)
- Reduction to Practice: KN042 Pod O BP005 (`commit 0696f31`)

**The Architectural Problem (enabling disclosure):**

Under conventional AI agent memory management, an agent's canonical knowledge file (e.g.,
MEMORY.md) must contain entries for every topic the agent might need to address in any
session. As canonical knowledge grows, the memory file grows proportionally. A 200-topic
canon requires a 200-entry file; a 1,000-topic canon requires a 1,000-entry file. Context
cost grows linearly with topic count regardless of task relevance.

**The Substrate-Routed Solution:**

Substrate-Routed Memory Expansion separates the memory INDEX from memory CONTENT:

```
MEMORY.md (index layer — stays small):
  - topic: "Federation Library L5 algorithm"
    trigger_class: vocabulary
    trigger_pattern: "Federation Library|L5|97\\.2%"
    content_ref: federation_library_l5
    (full content NOT loaded — wrasse delivers on trigger match)

  - topic: "Prov 16 filing details"
    trigger_class: vocabulary
    trigger_pattern: "Prov 16|provisional 16|KN071"
    content_ref: prov_16_details
    (full content NOT loaded)
```

The index entry is ≤5 lines. The full topic content (which may be 50-500 lines) lives in
the Wrasse substrate. When a session's incoming context matches a trigger pattern, the Wrasse
Scribe (#2332, already in Prov 15) delivers the full topic content — and only that topic's
content — into the session context.

**Measured empirical result (KN042 Pod O BP005, `commit 0696f31`):**
- MEMORY.md compacted: 405 lines → 141 lines (65% reduction)
- Substrate canon simultaneously grew by 60+ Eblets (no ceiling hit)
- Trigger test suite: 8/8 triggers fired correctly on test context injection
- Full test suite: 33/33 tests passing at commit
- Context cost profile: session pays for relevant-topics-only, not total-canon

**The scaling invariant:** As canon grows from 60 to 600 to 6,000 Eblets, MEMORY.md index
size grows by one 5-line entry per new topic — not by the full topic content. Total index
size approaches O(N × 5) lines, not O(N × full-topic-length). The ratio of index-size to
total-canon-size decreases monotonically as canon grows.

**Composition with Wrasse Pre-Injection (#2332, Prov 15):** Wrasse is the delivery mechanism.
Substrate-Routed Memory Expansion is the ARCHITECTURAL PATTERN — the decision to store
canonical topic content in the substrate (not inline in the index) and route topic delivery
through trigger-matched injection. Wrasse implements the trigger-matching and delivery
mechanics; Substrate-Routed Memory Expansion is the system architecture that makes Wrasse's
delivery pattern the load-bearing memory model for the entire substrate.

**Scope wedge — variation coverage (Pawn Q1 defense):**

Variation 1: Index file is not MEMORY.md but any agent-configuration file serving the
memory-index role (system prompt, initialization file, settings YAML, etc.)

Variation 2: Trigger matching is not regex-first but embedding-based, BM25-based, or
hybrid — the routing mechanism is implementation-variable; the O(relevant-topics-per-task)
scaling property is the architectural invariant.

Variation 3: Content delivery is not pre-injection but retrieval-on-demand — agent queries
substrate for topic content mid-session rather than receiving pre-injected block. The
index-content separation and O(relevant-topics) cost profile are preserved.

Variation 4: Index lives in-substrate (Pheromone/JSONL/vector DB) rather than in a
human-readable flat file, with programmatic topic resolution at session-open.

### Section II — Ring of Three Golden Eblets (#2337)

**Innovation Title:** Ring of Three Golden Eblets: Three-Component Top-of-Substrate Authority
Structure with Non-Overrideable Marked Exception Enforcement

**Classification:**
- Innovation Number: #2337 (candidate)
- Crown Jewel Candidate: YES
- Patent Provisional: Prov 16
- Conceived by: Jonathan Jones (Founder & General Manager)
- Reduction to Practice: BP005 base ratification — 3 Golden Eblets at `~/.claude/state/eblets/CANON/GOLDEN/` (282 combined lines) + federation canon `project_ring_of_three_golden_eblets_deck_card_medallion_federation_canon.md` (450+ lines)

**The Authority Architecture:**

The Ring of Three Golden Eblets is a three-component authority structure forming the top
layer of every substrate deployment:

```
GOLDEN EBLET 1 — Canon:
  Content: Inviolable principles. These are not adjustable by any downstream authority.
  Enforcement: Marked Exception — LB-source artifact QR ALWAYS routes to Canon.
  Override status: NON-OVERRIDEABLE by any downstream layer.

GOLDEN EBLET 2 — Platform Rules:
  Content: Platform-wide operational directives. Govern ALL agents on the platform.
  Enforcement: Applies uniformly across all projects on the platform.
  Override status: NON-OVERRIDEABLE at project layer (Part A of each platform rule).

GOLDEN EBLET 3 — Project Rules:
  Content: Project-specific operational directives within Canon + Platform Rule bounds.
  Enforcement: Applies within the project scope; cannot override Eblet 1 or 2.
  Override status: Adjustable only within the headroom left by Eblets 1 + 2.
```

**The Marked Exception mechanism:** When an agent receives an LB-source artifact (identified
by QR code, Call Sign, or canonical provenance marker), the agent's context-loading sequence
MUST route to Canon (Eblet 1) regardless of any downstream rule modification. This
non-overrideable routing is the Marked Exception: the project owner cannot configure their
Project Rules to bypass Canon routing for LB-source artifacts. This is structurally enforced
(not policy-enforced) — the Wrasse trigger-class for LB-source artifacts points unconditionally
to Eblet 1.

**Recursive instantiation:** The Ring of Three is not a fixed three-layer structure. At each
successive authority layer (L2 dual-entity, L3 project owner, L4 sub-project, L5 member),
the authority layer instantiates its own Ring of Three Golden Eblets for its scope. The
parent's Canon (Eblet 1) propagates non-overrideable through every child Ring. The child
adds its own Part A (non-overrideable within the child scope) and Part B (adjustable within
the child scope). This recursive composition is the subject of #2338 Multi-Layer Authority
Recursion.

**Scope wedge — variation coverage:**
- Variation: Ring of Two (Canon + Unified Rules) — reduced composition with same enforcement
- Variation: Ring of Four or N — extended composition; same Marked Exception mechanism applies
- Variation: Non-file-based implementation — Ring expressed as database records, API policy
  objects, smart contract rules, or equivalent structured artifacts
- Variation: Multi-substrate Ring composition — Ring instantiated across vendor-heterogeneous
  agent environments with cross-vendor Marked Exception enforcement

### Section VIII — Augur Living Gate (#2343)

**Innovation Title:** Augur Living Gate: Substrate-Event-Driven Freshness Gate Replacing
Clock-TTL with Pheromone-Write-Timestamp Comparison

**Classification:**
- Innovation Number: #2343 (candidate)
- Patent Provisional: Prov 16
- Reduction to Practice: KN038 Pod N BP004 (`commit 98d1745`) — 18/18 tests + T12 empirical PROOF

**The Architectural Innovation:**

Conventional AI substrate freshness mechanisms use clock-TTL: "re-derive this canonical fact
every N minutes/hours/days." The TTL fires even when nothing in the substrate has changed,
generating unnecessary re-derivation cost.

The Augur Living Gate replaces clock-TTL with substrate-event-driven gating:

```python
def augur_gate_check(topic, last_consult_ts, pheromone_client):
    """
    Gate fires ONLY when substrate state has actually changed.
    """
    last_write_ts = pheromone_client.get_last_write_ts(topic)

    if last_write_ts is None:
        # No Pheromone record available — hard ceiling fallback
        hours_since = (now() - last_consult_ts).total_seconds() / 3600
        return hours_since > HARD_CEILING_HOURS  # e.g., 72 hours

    # Gate fires only if substrate was written AFTER last consult
    return last_write_ts > last_consult_ts
```

**T12 Empirical Proof (KN038, `commit 98d1745`):**
Test T12 demonstrates the key property: when no Pheromone write has occurred since last
consult, the gate remains OPEN (allow re-use of prior derivation) regardless of how much
clock time has elapsed. This proves the gate responds to substrate state, not clock state.

**Extension to all Augur gates (KN043 Pod P BP005):** The Augur Living Gate pattern was
generalized from the single-topic case to ALL Augur gates across the substrate — any gate
that previously used clock-TTL was migrated to Pheromone-write-timestamp comparison.

### Section X — The Shadow (Persistent Background Watcher Daemon) (#2345)

**Innovation Title:** The Shadow: Persistent Background Watcher Daemon Spawned at
AI Agent SessionStart and Killed at SessionEnd

**Classification:**
- Innovation Number: #2345 (candidate)
- Patent Provisional: Prov 16
- Reduction to Practice: KN037 v2 Pod M BP004 (`commit ba2131f`) — 13/13 tests + live smoke test green

**Architecture:**

The Shadow is a persistent OS-level background process spawned by the SessionStart hook
of a persistent-substrate AI agent. It runs in parallel with the agent's session, monitoring
substrate state and capturing cross-monitor context.

```bash
# SessionStart hook fires:
spawn_shadow() {
    python bishop_shadow_watcher.py \
        --session-id "$SESSION_ID" \
        --output-dir "$SHADOW_OUTPUT_DIR" \
        --monitor-interval 5s \
        &
    echo $! > "$SHADOW_PID_FILE"
}

# SessionEnd hook fires:
kill_shadow() {
    if [ -f "$SHADOW_PID_FILE" ]; then
        kill $(cat "$SHADOW_PID_FILE")
        rm "$SHADOW_PID_FILE"
    fi
}
```

**Brand-class wrapper role:** The Shadow is the canonical brand-class for ALL persistent
background tasks in the substrate. Any long-running monitoring process, watcher daemon,
or cross-session state-capture process is a Shadow. This provides a unified vocabulary and
a common implementation pattern (spawn-on-start / kill-on-end / PID-file tracking).

**Why the Shadow requires the persistent-substrate agent class (#2342):** The SessionStart
and SessionEnd hooks that spawn and kill The Shadow are only available to persistent-substrate
agents (Bishop class). Per-session agents (Knight class) do not have hook access and therefore
cannot maintain Shadows. This is the concrete technical reason for the "Knight-Has-No-Shadows"
architectural discipline.

### Section XI — Augur PostToolUse Audit + Stone Tablet Supersede Pattern (#2346)

**Innovation Title:** PostToolUse Audit Architecture with Stone Tablet Supersede Pattern:
Non-Blocking Discipline Enforcement via Append-Only Violation Records

**Classification:**
- Innovation Number: #2346 (candidate)
- Patent Provisional: Prov 16
- Reduction to Practice: KN043 Pod P BP005 (`commit 7701c84`) — 8/8 tests; Augur-Pricing PreToolUse blocking ENDED; `bishop_augur_post_audit.py` operational

**The PreToolUse-to-PostToolUse Migration:**

PreToolUse-blocking discipline gates interrupt agent execution: the agent's tool call is
blocked until the gate clears. For gates that require external verification (Augur-Pricing,
for example), blocking introduces latency, context loss risk, and agent timeout exposure.

The PostToolUse Audit pattern converts blocking gates to non-blocking audit + Stone Tablet
supersede:

```
BEFORE (PreToolUse-blocking):
  Agent wants to write pricing content →
  Gate fires →
  Gate blocks tool execution →
  Gate checks Augur-Pricing rules →
  If violation: tool call blocked indefinitely
  If clear: tool call proceeds

AFTER (PostToolUse-audit + supersede):
  Agent writes content (tool call proceeds immediately) →
  PostToolUse hook fires →
  Audit reads the written content →
  If violation: audit writes violation record as Stone Tablet;
                writes supersede-pointer file pointing to violation record;
                Bishop reconciliation queue notified
  If clear: audit writes PASS record as Stone Tablet
```

The supersede-pointer file is the architectural innovation: it converts a blocking violation
into a non-blocking audit trail that the Bishop (persistent-substrate agent) reconciles at
its next session boundary. The agent is not interrupted; the discipline record is preserved;
the Bishop has full provenance for reconciliation.
