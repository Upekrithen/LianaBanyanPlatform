## §1 CANONICAL DEFINITION

**Hard Candy Eblet** = an AI agent configuration packaged as a canonical, shareable Eblet. A Hard Candy Eblet contains the full operational configuration for a Stitchpunk Sock Puppet AI (Bishop, Knight, Pawn, Rook, or any member-created variant) — wrapped in the Eblet format so it can be versioned, curated into a MENU, shared cooperatively, and instantiated by others.

"Hard Candy" is an intentional naming: something that lasts, has a defined shape, can be unwrapped and used. Hard Candies don't expire quickly (they are ANCHOR-decay-class) and they have a recognizable form (canonical configuration). When a member receives a Hard Candy Eblet, they know exactly what they are getting.

## §2 MOTIVATION / FOUNDER VISION

The Founder's long-term vision (articulated BP058, the Mnemosyne endpoint): a member speaks to Ollama, and the Ollama routing layer allocates the work to the appropriate Stitchpunk AI based on its loaded Hard Candy configuration. The member doesn't manage which AI does what — the configuration does. The Hard Candy Eblet is what makes each Stitchpunk's behavior deterministic, transferable, and cooperative-class shareable.

This vision requires:
1. Each Stitchpunk AI has a canonical Hard Candy configuration Eblet
2. The configuration is available in Mnemosyne (not locked to one machine or session)
3. Members who want to instantiate their own Bishop/Knight/Pawn variant can unwrap the Hard Candy and run it
4. Members can share their own custom configurations — extending the cooperative-class intelligence ecosystem

The Founder is explicit: *"This is exactly why Mnemosyne exists, but we can't use it that way yet. WE need to get there. Fast."* Hard Candy Eblets are the structural foundation of that vision.

### §3.3 Shareable Mechanics

When the Founder (or any member) marks a Hard Candy Eblet as shareable via MENU:
1. It appears in Helm categorical view under "Hard Candies" food-class
2. Any member can fork it into their own MENU with full cooperative-class attribution
3. Forkers customize the configuration for their own persona/use case
4. Attribution chain is permanent: "Forked from Founder's Bishop Hard Candy v2"
5. Forked variants are not "the same AI" — they are the member's own Stitchpunk running on the member's own substrate, configured via the forked template

### §3.4 Ollama Vision Composition

The full Ollama allocation vision (BP058):
- Member speaks a request to the Mnemosyne interface
- Ollama routing layer reads the loaded Hard Candy configurations
- Request is allocated to the Stitchpunk whose Hard Candy configuration best matches the work-type
- Each Stitchpunk executes per its configuration
- Results flow back to the member through Mnemosyne

Hard Candy Eblets are the declaration layer for this routing architecture. Without Hard Candies, the routing layer has no configuration to read.

## §5 CONSTRAINTS + ANTI-PATTERNS

**Hard Candy Eblet is NOT:**
- A Sock Puppet in the negative sense (impersonation, deception, shadow identity without disclosure). Hard Candy = cooperative-class AI persona, owned by a real cooperative member, transparently attributed. Sock Puppet = Shadow E-Giant class (different lineage, different use).
- A runtime execution environment (the Hard Candy is the configuration, not the running AI instance — the AI instance is spun up separately using the config)
- Platform-locked (Hard Candy Eblets must be portable — member can run their configured Stitchpunk via any compatible model host, not only the Founder's default)
- Anonymous (every Hard Candy has an owner, a version, and attribution chain — cooperative-class attribution discipline enforced)
- The same as a Shadow E-Giant (Shadow E-Giants are platform-dispatch constructs; Hard Candies are member-owned cooperative-class configurations)

**Anti-patterns:**
- Treating a Hard Candy Eblet as the running AI (the Eblet is configuration data; the running AI is the instantiated agent)
- Sharing a Hard Candy without attribution tracking — every fork preserves the full attribution chain
- Building Hard Candies that include API keys or secrets — configuration is behavioral, not credential
- Creating Hard Candies that override cooperative-class behavioral rules — a Hard Candy may customize but may not strip Trinity Rules or Anti-Hype disciplines from the canonical Stitchpunk configs
