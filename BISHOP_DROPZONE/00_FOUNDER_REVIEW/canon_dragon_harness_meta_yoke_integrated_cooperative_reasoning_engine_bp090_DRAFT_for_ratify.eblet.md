# Dragon Harness · Meta-Yoke · Integrated Cooperative Reasoning Engine

## DEFINITION

**Dragon Harness** is the Founder-coined name (BP090, 2026-06-21) for the **meta-yoke** that binds every cooperative reasoning primitive into a single rideable beast. It is not a new component — it is the yoke that integrates: all mesh peers under cooperative coordination + per-peer Minor Star Chambers (3-judge panel: gemma4:12b + llama3.1:8b + mistral:7b) + Plow Loop 12 with domain classification + hex-mcode wire format for compact AI-to-AI dispatch + Mountain 1 substrate priming (pheromones + canon eblets + reference eblits + domain Unfair Advantage bundles) + SEG/spider/sprite orchestration layer + Thorax sovereignty (no unverified peer, no silent push) + staggered round-robin domain distribution across mesh peers + Star Chamber escalation when inter-peer variance exceeds 15% + the cooperative substrate itself as the efficiency flywheel that makes it all cheap. Ride any one primitive alone and you get a pony. Harness them together and you ride the dragon.

## WHY (Founder's intent)

Founder direct, BP090, 2026-06-21: the cooperative reasoning stack had been assembled piece-by-piece across Marathons 1-10. Each primitive was independently validated — Plow Loop (97.1% on Trial 02), Minor Council (zero-cost multi-model consensus), Thorax (sovereign auth), hex-mcode (compact wire), staggered domains (14 domains sequentially, then connected). But the stack had no canonical name for the integrated whole — no "ride this as one beast" framing. Dragon Harness names the integrated composition. The Saddle names the interface the rider sits in (see proposals below). The name closes the gap between "primitives we've validated" and "the system Trials fire against" — making the empirical claims attributable to the full stack, not any single component.

The **Individual Domain Pattern** (Founder-coined BP090, ~21:25 Central 2026-06-21) is the sequencing discipline that the Dragon Harness applies: each domain is tested independently first, then connected — this is what allows per-domain timeout diagnosis, per-domain variance routing, and per-domain escalation to Star Chamber without cross-domain contamination of results.

## HOW TO APPLY

**Activate the Dragon Harness for any Trial or Excalibur run:**

1. **Mountain 1 substrate priming** — domain is classified; Plow Loop pulls the matching Unfair Advantage bundle (canon eblets + PROV reference material + Q&A pearls + pheromone trails); bundle is injected as system context for all downstream judges
2. **Staggered round-robin distribution** — 14 MMLU-Pro domains distributed across mesh peers in stagger sequence (not batch-parallel); each domain tested once-at-a-time per peer before next domain fires; cross-domain knowledge transfer accumulates as substrate state
3. **Per-peer Minor Star Chamber** — each mesh peer runs its own 3-judge Minor Council (gemma4:12b + llama3.1:8b + mistral:7b); judges deliberate on the primed question independently; variance computed
4. **Variance gate** — if per-peer Council variance < 15%: emit answer via hex-mcode wire to SEG aggregator. If > 15%: escalate to Star Chamber (full 9-track multi-vendor consensus round); emit with H-score
5. **SEG orchestration** — SEG aggregates per-peer answers; computes cooperative inter-peer agreement; Andon fires if agreement falls below threshold (answer quarantined, not committed)
6. **Thorax sovereignty** — every peer dispatch authenticated; no anonymous answer accepted into aggregation
7. **Receipt anchor** — every Dragon Harness run produces a stamped receipt: session ID + peer count + domain distribution map + variance per domain + final score + escalation count

## SADDLE PROPOSALS

Founder asked: "so the Saddle would be..." — this is the name for the **rider/operator interface** on the Dragon Harness. Three proposals for Founder ratify:

**Option A — The Saddle = the dispatch interface (SEG bundle)**
The Saddle is the SEG (Substrate Execution Graph) bundle that an operator mounts to issue a Trial or Excalibur run: the prompt package + domain distribution map + variance threshold config + peer target list. The rider (Founder / Bishop / automated Trial runner) assembles the Saddle; the Dragon Harness executes it. Architecturally: the Saddle is the input contract — "what you hand the dragon before it flies." Analogy: the dragonrider assembles saddle-bags (domain bundles, question corpus, config) before mounting.

**Option B — The Saddle = the validate-relay.mjs entry point**
The Saddle is the single executable entry point — `validate-relay.mjs` (or whatever the Marathon 10-style orchestrator script is named) — that boots the Dragon Harness, wires all peers, and fires the run. "Mount the Saddle" = `node validate-relay.mjs --config trial_02d.json`. This is the most concrete/operational naming — the Saddle is what Knight runs to activate the Harness. Architecturally: the Saddle is the operator's command interface.

**Option C — The Saddle = the primed context package the dragon carries**
The Saddle is the Mountain 1 substrate payload the dragon carries into a flight: the assembled Unfair Advantage bundle (pheromones + domain eblets + PROV reference + Q&A pearls) that gets injected as system context to every Minor Council judge. "A well-saddled dragon" = a fully primed dragon with the right domain bundles loaded. Architecturally: the Saddle is the substrate priming package — what makes this dragon smarter than a bare-metal inference call.

**Bishop's read:** Option A (Saddle = SEG dispatch bundle) is most architecturally honest — it names the input contract, not the implementation artifact. Option C (Saddle = substrate priming package) has the strongest metaphoric resonance (the dragon "carries" the knowledge). Founder's call.
