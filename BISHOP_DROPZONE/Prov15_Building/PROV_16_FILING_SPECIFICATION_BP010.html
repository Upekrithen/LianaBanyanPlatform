Description

The Iron Tablet is the successor primitive to the Stone Tablet (#2327) for the multi-organism Iron-E-Giant Federation substrate era. While the Stone Tablet provides per-organism append-only lossless preservation, the Iron Tablet fuses three substrate functions that must operate atomically in a multi-organism federated environment:

  1. Append-only hash-anchored ledger — the substrate ledger for the Iron-E-Giant Federation; each append carries a cryptographic hash linking it to the prior append, creating an immutable ordered chain verifiable by any organism in the Federation
  2. Wrasse-routed content-addressed active memory — the Iron Tablet IS the content-addressed memory substrate for active cross-organism work; Wrasse routing delivers Iron Tablet content by topic and task-class without requiring organisms to hold complete ledger copies
  3. Scribe-mutex-serialized federation-coordinated multi-organism concurrent-safe write authority — the Stone-mutex serialization layer that prevents concurrent-write conflicts across 5+ simultaneously-running organisms; ConcurrencyConflict detection on divergent writes; last-writer-wins Eblet discipline with ledger-survives-Eblet-deletion recovery (T5 receipt)

Empirical receipt (KN089, commit 8380701): - 5-organism concurrent-write scenario: Stone-mutex serialization pass; ConcurrencyConflict correctly emitted on 4/5 divergent writes; T5 ledger-survives-Eblet-deletion recovery verified - 5/5 new tests + 161 total green (zero failures) - 5 modules: iron_tablet/ — ledger.py, memory.py, write_authority.py, mutex.py, recovery.py

A method for fused atomic substrate primitive operations in a cooperative-AI multi-organism federation, comprising:

  1. Maintaining a per-organism append-only hash-anchored ledger where each record carries a cryptographic hash of the prior record, making the ledger tamper-evident and verifiable by any peer organism in the Federation;
  2. Providing content-addressed active memory via Wrasse-routed topic delivery, where organisms receive Iron Tablet content bound to their current task-class without holding complete ledger copies — decoupling memory delivery cost from ledger size;
  3. Serializing all write operations through a scribe-mutex, where each write attempt claims the mutex, checks for concurrent-write conflicts against the current ledger state, and emits a ConcurrencyConflict record when divergent writes are detected;
  4. Applying last-writer-wins Eblet discipline for conflict resolution under a defined quorum threshold, preserving the ledger integrity even when individual Eblet files are deleted (ledger-survives-Eblet-deletion recovery);
  5. Distributing write-confirmed Iron Tablet records to all registered Federation organisms via the Pheromone Substrate (#2317) as topic-keyed cross-organism memory.
