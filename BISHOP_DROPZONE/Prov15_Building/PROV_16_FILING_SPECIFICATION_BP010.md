## A1. Iron Tablet Primitive (KN089)

### Innovation Title
**Iron Tablet: Fused Atomic Substrate Primitive Combining Append-Only Hash-Anchored Ledger,
Wrasse-Routed Content-Addressed Active Memory, and Scribe-Mutex-Serialized Federation-Coordinated
Multi-Organism Concurrent-Safe Write Authority**

### Classification
- **Innovation Number:** #2NNN-A1 (CANDIDATE — Founder retains assignment authority)
- **Crown Jewel Candidate:** YES — fused atomic primitive simultaneously providing three substrate
  functions (ledger + memory + write authority) that prior art provides only separately; composes
  with Pheromone Substrate (#2317) as fused successor primitive for the Iron-E-Giant Federation era
- **Patent Provisional:** Prov 16
- **Date of Conception:** BP011 (2026-04-20 era)
- **Session of Origin:** BP011
- **Conceived by:** Jonathan Jones (Founder & General Manager)
- **Reduction to Practice:** Knight commit `8380701`; 5 modules in `librarian-mcp/src/iron_tablet/`;
  5/5 new tests + 161 total green; 5-organism concurrent-write Shutterbug receipt
- **Cooperative Defensive Patent Pledge:** Filed under #2260

### Description

The **Iron Tablet** is the successor primitive to the Stone Tablet (#2327) for the multi-organism
Iron-E-Giant Federation substrate era. While the Stone Tablet provides per-organism append-only
lossless preservation, the Iron Tablet fuses three substrate functions that must operate atomically
in a multi-organism federated environment:

1. **Append-only hash-anchored ledger** — the substrate ledger for the Iron-E-Giant Federation;
   each append carries a cryptographic hash linking it to the prior append, creating an
   immutable ordered chain verifiable by any organism in the Federation
2. **Wrasse-routed content-addressed active memory** — the Iron Tablet IS the content-addressed
   memory substrate for active cross-organism work; Wrasse routing delivers Iron Tablet content
   by topic and task-class without requiring organisms to hold complete ledger copies
3. **Scribe-mutex-serialized federation-coordinated multi-organism concurrent-safe write authority**
   — the Stone-mutex serialization layer that prevents concurrent-write conflicts across 5+
   simultaneously-running organisms; ConcurrencyConflict detection on divergent writes;
   last-writer-wins Eblet discipline with ledger-survives-Eblet-deletion recovery (T5 receipt)

**Empirical receipt (KN089, commit `8380701`):**
- 5-organism concurrent-write scenario: Stone-mutex serialization pass; ConcurrencyConflict
  correctly emitted on 4/5 divergent writes; T5 ledger-survives-Eblet-deletion recovery verified
- 5/5 new tests + 161 total green (zero failures)
- 5 modules: `iron_tablet/` — ledger.py, memory.py, write_authority.py, mutex.py, recovery.py

### Method Claim (Independent)

A method for fused atomic substrate primitive operations in a cooperative-AI multi-organism
federation, comprising:

1. Maintaining a per-organism append-only hash-anchored ledger where each record carries a
   cryptographic hash of the prior record, making the ledger tamper-evident and verifiable
   by any peer organism in the Federation;
2. Providing content-addressed active memory via Wrasse-routed topic delivery, where organisms
   receive Iron Tablet content bound to their current task-class without holding complete ledger
   copies — decoupling memory delivery cost from ledger size;
3. Serializing all write operations through a scribe-mutex, where each write attempt claims the
   mutex, checks for concurrent-write conflicts against the current ledger state, and emits a
   ConcurrencyConflict record when divergent writes are detected;
4. Applying last-writer-wins Eblet discipline for conflict resolution under a defined quorum
   threshold, preserving the ledger integrity even when individual Eblet files are deleted
   (ledger-survives-Eblet-deletion recovery);
5. Distributing write-confirmed Iron Tablet records to all registered Federation organisms via
   the Pheromone Substrate (#2317) as topic-keyed cross-organism memory.

### Dependent Claims

6. The method of claim 1, wherein the hash-chain is verified at organism spawn-time by checking
   the most recent N records before accepting new appends, providing startup-integrity assurance.
7. The method of claim 1, wherein ConcurrencyConflict records are themselves appended to the
   ledger as first-class records, creating an auditable conflict history.
8. The method of claim 3, wherein the scribe-mutex carries organism-identity metadata, enabling
   post-hoc attribution of each write to its originating organism.
9. The method of claim 4, wherein the ledger-survives-Eblet-deletion recovery reconstructs
   organism-local Eblet state from the hash-chain record when an Eblet file is found missing.
10. The method of claim 1, further comprising a Bounty #7 community-empirical-tuning pathway
    (see A6) wherein organism operators may submit bounded-range configuration adjustments to
    Iron Tablet operational parameters via Founder-ratified YAML config.

### System Claim

A system for Iron Tablet fused atomic substrate operations, comprising:
- A **ledger module** (`ledger.py`) implementing append-only hash-chain records at a canonical
  path within the Iron-E-Giant Federation's substrate directory
- A **memory module** (`memory.py`) implementing Wrasse-routed content-addressed active memory
  delivery, partitioned by task-class and organism identity
- A **write authority module** (`write_authority.py`) providing scribe-mutex serialization with
  ConcurrencyConflict detection and last-writer-wins resolution
- A **mutex module** (`mutex.py`) implementing organism-aware lock management with timeout
  fallback for Federation-scale concurrent operations
- A **recovery module** (`recovery.py`) implementing ledger-survives-Eblet-deletion hash-chain

### Innovation Title
**In-Concert: Multi-Organism Coordination Protocol with Pheromone-Anchored Cross-Organism
Decision-Routing, Furnace Cross-Organism Verification, Stone-Tablet-Trumps-Conflict Audit,

### Classification
- **Innovation Number:** #2NNN-A3 (CANDIDATE)

- **Conceived by:** Jonathan Jones (Founder & General Manager)

  5/5 cross-org scenarios PASS; 170 total tests (zero failures)
- **Cooperative Defensive Patent Pledge:** Filed under #2260

### Description

The **In-Concert Protocol** is the multi-organism coordination layer of the Iron-E-Giant
Federation. When multiple organisms (Bishop instances, Knight instances, Pawn instances,
Shadow-E-Giant daemons) operate simultaneously on shared substrate work, the in-concert
protocol coordinates their decision-making and conflict-resolution without requiring a
centralized authority.

**5 empirically-receipted cross-organism scenarios (KN091):**
- (a) Cross-org Furnace verification at 0.65 threshold — organisms propose a substrate-state
  claim; Furnace cross-org verification requires quorum ≥0.65 of Federation organisms to
  verify before the claim is canonized
- (b) ConflictReport emission — when organisms produce divergent outputs on the same task,
  the in-concert protocol emits a ConflictReport as an Iron Tablet record rather than silently
  discarding either output
- (c) Stone-Tablet-trumps-conflict audit — canonical Stone Tablet records always supersede
  in-flight organism disagreements; the audit layer ensures Stone Tablet authority is preserved
  against quorum-based overrides
- (d) Bishop+Knight cohort overlap-refresh — when a Bishop context-window refreshes mid-session
  while a Knight session is in-flight, the overlap-refresh protocol preserves all in-flight
  state across the cohort boundary (zero state-loss)
- (e) Graceful degradation — when fewer than quorum organisms are available, the in-concert
  protocol falls back to single-organism execution rather than halting

### Method Claim (Independent)

A method for multi-organism coordination in a cooperative-AI federation, comprising:

1. Routing cross-organism decisions via the Pheromone Substrate (#2317) as first-class decision

   ephemeral messages;
2. Requiring Furnace cross-organism verification at a configurable quorum threshold (default 0.65)

4. Applying Stone-Tablet-trumps-conflict audit: when a canonized Stone Tablet record exists on

5. Executing cohort overlap-refresh on parent-organism context-window reset: all in-flight
   state is checkpointed to Iron Tablet before the refresh, and restored with verified
   hash-chain continuity afterward, achieving zero state-loss;
6. Falling back to single-organism execution (graceful degradation) when available organism

**Cross-Vendor Peer-Organism Gateway: Integrating Non-MCP-Native AI Vendor as Federation

### Description

The **Pawn-Librarian Gateway** integrates the Pawn agent class (Perplexity API — a non-MCP-native
vendor) as a fully-empowered peer organism in the Iron-E-Giant Federation.

**The integration architecture:**

**Shadow-α proxy as ground-truth intermediary:** The Pawn agent (Perplexity API) cannot
directly invoke MCP tools. Instead, it communicates with a Shadow-α daemon (a Shadow-E-Giant
organism in the alpha LIGHTHOUSE position) via an HTTP API surface that the Shadow translates
into MCP-native tool calls. The Shadow acts as a ground-truth proxy — Pawn's queries arrive
as HTTP requests; Shadow translates, executes MCP tools, and returns structured responses.

**Safe-tool-list enforcement:** Five categories of tools are hard-denied to cross-vendor peers:
write to canonical Stone Tablet, canonical-state-flip, production deployment, USPTO submission,
and Founder-identity impersonation. The safe-tool-list is enforced at the gateway layer;
Pawn cannot request exemptions via API.

**Bishop-VETO power:** Any cross-vendor peer write that would modify a shared Federation substrate
record requires Bishop VETO-pass before execution. Bishop reads the write intent, validates
against current substrate state, and either passes (write proceeds) or VETOs (write blocked,
ConflictReport emitted).

**Empirical receipt (KN092, T4 BP011-failure-mode-closure):**
Pawn successfully read 6,285 chars of `behemoth-reborn.md` via the Shadow-α proxy gateway;
cross-vendor 5-phase Handshake completed; 5 hard-denied tools confirmed blocked.

**PostToolUse audit architecture:**

1. The original Stone Tablet record from the tool call is preserved (immutability maintained)
2. A supersede-pointer record is written to a canon file, flagging the original record as
   "superseded pending Bishop reconciliation"
3. Bishop reviews the supersede queue at the next session and takes corrective action if needed

This pattern converts a hard block (flow disruption) into a soft flag (audit + queue) while
preserving the full discipline record.

4. On VIOLATION FLAG: (a) preserving the original Stone Tablet record from the tool call
