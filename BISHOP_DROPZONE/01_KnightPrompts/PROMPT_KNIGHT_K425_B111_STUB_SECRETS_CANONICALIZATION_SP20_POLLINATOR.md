## Scope — two distinct workstreams, related

### Workstream A — Secrets canonicalization

**Problem (B111 observation, Founder-raised):** API keys currently live in TWO places — LockBox SDS.env (local file, read by SP-18 / Cursor / Knight scripts) AND Supabase Vault (`vault.secrets` table, read by Edge Functions / platform runtime). The dual storage is accidental, not designed. Tools don't share a single read-path, so adding a new vendor means two inserts.

**Goal:** One canonical source (Supabase Vault), one read-path convention, one mirror policy.

**Deliverables:**
1. Audit: full list of every secret currently in SDS.env, Supabase Vault, and any third storage location (Cursor `mcp.json`, FireBase Login.txt, etc.). Produce `SECRETS_AUDIT_B111.md` in `03_BishopHandoffs/` — name / purpose / which tools read it / current storage / desired storage.
2. **Supabase Vault as canonical.** All production secrets stored there. Rotatable, scoped, auditable, survives machine loss.
3. **SDS.env as auto-generated mirror** for local Python/Knight/Cursor scripts that can't easily hit Supabase at call time. Mirror generated by a script `scripts/sync-sds-from-vault.py` that pulls from Vault and writes SDS.env. Run on demand + pre-commit hook + CI.
4. **One canonical env-var namespace.** `ANTHROPIC_API_KEY`, `GOOGLE_API_KEY`, `OPENAI_API_KEY`, `PERPLEXITY_API_KEY`, `SUPABASE_URL`, `SUPABASE_SERVICE_KEY`, `SUPABASE_ANON_KEY`, etc. Fix any existing scripts that use non-standard names.
5. **Deprecate `DOUBLESECRETBACKUP.env`** if it exists in the audit (Founder-raised). Either fold its contents into Vault+SDS.env canonical path, or document clearly what it's for and why it's separate.
6. **Documentation:** `docs/SECRETS.md` in `librarian-mcp-public/` repo — public-friendly version of the secrets-management convention (names only, no values, so community testers of the Eyewitness Program have clear guidance for their own env-var setup).

**Risks to watch for:**
- Supabase Vault rotation behavior. Confirm keys survive key-rotation events on the Supabase side.
- Mirror script should never COMMIT the SDS.env file. Enforce via `.gitignore` verification.
- If a key is used by BOTH Bishop (via SDS.env) AND an Edge Function (via Vault), the mirror direction matters — Vault is authoritative, SDS.env is generated. Never generate Vault from SDS.env.

### Workstream B — SP-20 Pollinator (new Stitchpunk)

**Founder-approved B111:** *"Pollination Stitchpunk approved (SP-20). Include it with what is now K425 for you to make."*

**Concept:** Pollinator takes ratified canonical sentences / phrases / concepts (e.g., the B111 Bishop-Founder collaboration sentence: *"The mistake was mine for assuming; the correction was quick because you were specific"*) and propagates them across target canonical surfaces, keeping the surfaces aligned. Unlike SP-15 (which extracts decisions from transcripts) or SP-16 (which recombines canonical elements into new drafts), SP-20 is propagation-forward only: a sentence ratified in one place should appear, unchanged, in every surface that thematically requires it.

**Design requirements:**
1. **Canonical source.** SP-20 reads from `memory/` (Bishop's memory vault) and `BISHOP_DROPZONE/00_FOUNDER_REVIEW/` ratified docs. New memory files with a `pollinate_to:` frontmatter field are eligible for propagation.
2. **Target surfaces (configurable per pollination item):**
   - AI Tuning documentation (location TBD — find or create `docs/ai_tuning/` in `LianaBanyanPlatform/`)
   - Platform blueprints (find or create `docs/blueprints/`)
   - Founder Stats / scoreboards (find the existing surface; likely `src/pages/FounderStatsPage.tsx` or similar — audit needed)
   - Relevant Crown Letters / Papers (pollinate only when thematically appropriate; SP-20 should NOT auto-modify letters — it should write a *pollination request* artifact that Bishop/Knight reviews before propagating)
   - Session-start Bishop briefings (new Bishop sessions see the pollinated canon as part of their orientation context)
3. **Idempotency.** Running SP-20 twice on the same ratified sentence should not produce duplicates. Track "already pollinated to X" state per sentence per surface.
4. **Human review gate before propagation into letters.** SP-20 can auto-pollinate to docs/blueprints/scoreboards; it must NOT auto-pollinate into Crown Letters or public-facing papers without Bishop review. Those are Founder-voice surfaces; propagation requires a review step.
5. **Initial pollination targets (day-1 backlog):**
   - Bishop-Founder collaboration sentence (`memory/project_founder_bishop_collaboration_dynamic.md`)
   - Wellspring Model concept summary (`memory/project_rhetorical_keystones.md` Keystone #14)
   - Thermometer Keystone #16 "shows only you" — already propagated manually to NYT v2 + Doctorow V04 + Scott v014h; SP-20 should validate these are aligned and flag any drift.
   - Inversion Principle (B109 canonical finding) — pollinate into blueprints + AI Tuning as a design-method primitive.
   - Anachronism Principle (B110 canonical finding) — pollinate into blueprints.

**File structure:**
- `librarian-mcp/stitchpunks/sp20_pollinator.py` — the Stitchpunk
- `librarian-mcp/stitchpunks/data/pollination_state.json` — idempotency state
- `librarian-mcp/stitchpunks/data/pollination_backlog.md` — human-reviewable queue of pending pollinations
- CLI: `python -m stitchpunks.sp20_pollinator --dry-run` (default) / `--apply` (writes changes) / `--surface <name>` (limit scope)

Natural-pollination is non-directed (bees go where they want); SP-20 is directed (Bishop specifies target surfaces explicitly).
