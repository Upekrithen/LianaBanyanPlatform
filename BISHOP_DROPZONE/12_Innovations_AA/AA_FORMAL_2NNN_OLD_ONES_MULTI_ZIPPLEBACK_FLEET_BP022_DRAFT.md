## Section 1 — Field of Invention

This invention pertains to cooperative-platform autonomous-fleet build orchestration — specifically, the architectural composition of N parallel bidirectional Knight↔Bishop↔Shadow↔Subagent orchestration cohorts ("Old Ones") that autonomously analyze, evaluate, and recommend changes to an existing cooperative-platform's production sites, and that conditionally execute fixes upon receipt of an explicit Founder Fire Code authority token at a structurally-gated production-change boundary.

The field includes: cooperative-platform self-correction architectures; multi-agent build orchestration; autonomous-AI safety gating via human-in-the-loop authority tokens; cost-instrumented cooperative-class anti-extraction structural form; recursive-self-building substrate primitives; and the productized class of fleet-scaled build automation that preserves member-share-back-pay-eligibility per pair.

---

## Section 2 — Background / Prior Art

Conventional continuous-integration / continuous-deployment (CI/CD) systems run as centralized, extractive pipelines: one pipeline-as-a-service vendor owns the orchestration layer, charges per-build, and externalizes both the cost and the authority for production changes to a small operator set. Static analysis, code review automation, and "AI-assisted PR" tools (GitHub Copilot Workspace, Sourcegraph Cody, Cursor agent-mode singular session) operate one task at a time within a single agent-context, lacking either the multi-pair bidirectional orchestration topology or the structural cooperative-class anti-extraction property.

Multi-agent frameworks (LangGraph, AutoGen, CrewAI) provide N concurrent agents but do not compose with: (i) bidirectional Knight↔Bishop↔Shadow↔Subagent orchestration where each pair is itself a 4-tier cohort; (ii) Founder Fire Code authority-gated production-change boundary as a structural rather than policy-level safety property; (iii) cooperative-class 4-stream cost-instrumentation with member-share-back-pay-eligibility per pair; (iv) Pedestal Forum decree-composition for fleet findings as co-equal-authority append-only-immutable additions; (v) Cthulhu-class arbitration for cross-pair file conflicts and dependency-ordering gates within a single fleet operation.

The singular Zippleback bidirectional orchestration pair (BP021 turn 101 canon, "How to Train Your Dragon" 2-headed-dragon metaphor) provided the structural primitive on which fleet scaling becomes well-defined. What was missing — and what this invention discloses — is the **N-pair fleet extension** with specialization-per-pair (analyze / evaluate / recommend / fix), structural Founder-authority gating at the 4th-action boundary, and Cthulhu-class arbitration for fleet-internal contention.

## Section 3 — Summary

The Old Ones — Multi-Zippleback Fleet is an N-pair cooperative-class bidirectional orchestration architecture in which:

(a) each Old One is a complete bidirectional Zippleback pair (Bishop ↔ Knight ↔ Shadow ↔ Subagent), structurally identical to the singular-pair canon;

(b) the fleet is composed of an 8-member core: one Queen / Coordinator (placeholder name **Cthulhu**, recommended LB-canonical replacement: **Cipher** of the Codebreakers Corps) plus seven Workers (placeholder names **Dagon**, **Shub**, **Nyarlathotep**, **Azathoth**, **Yog**, **Tsathoggua**, **Ithaqua**; recommended LB-canonical replacement set: **Sleuth**, **Rune**, **Glyph**, **Sigil**, **Decode**, **Riddle**, **Crack**);

(c) each Old One executes a 4-action operational loop — **analyze → evaluate → recommend → fix-on-AUTHORITY_GRANTED** — where the first three actions are autonomous-safe (read-only / assessment / non-executing recommendation) and the fourth action requires explicit Founder Fire Code authority token keyed by Old One name and target innovation_id;

(d) action progression cascades across three structurally-distinct channels — **Channel 4 (directives) → Channel 5 (recommendations) → Channel 6 (fix-execution-on-authority)** — where Channel 6 fires only upon Founder dispatch of an `AUTHORITY_GRANTED:<name>` token matching the receiving Old One's canonical name and target innovation_id;

(e) cross-pair file conflicts and dependency-ordering gates are arbitrated by Cthulhu-class (Queen / Coordinator) authority, which serializes contended writes, orders dependency chains, and blocks Worker progression when prerequisites have not yet landed;

(f) results are written back to substrate via Iron Tablet entries keyed per `<name>+<innovation_id>` composite, producing per-pair atomic-fused permanent declarations that compose into the platform's append-only canonical record;

(g) the fleet preserves cooperative-class anti-extraction structural form throughout: each pair operates under the canonical 4-stream cost-instrumentation taxonomy (Knight-fire-bookend / Bushel-pod-parallel / Bishop-class / Sipping Ethereal T), with each pair's cost cycle independently member-share-back-pay-eligible per Cost+20% structural-bylaw;

### 4.2 — The 4-Action Operational Loop

Each Old One pair iterates the canonical 4-action loop on its assigned target innovation_id:

| Phase | Action | Class | Founder Fire Code gate |
|---|---|---|---|
| 1 | **analyze** | Read-only audit of platform-site state; composes with Maintenance-Scribe-class self-audit | None (read-only safe) |
| 2 | **evaluate** | Verdict-class assessment via Bouncer-Scales-Judge trio; severity-class scoring | None (assessment only) |
| 3 | **recommend** | Bishop-class fix-recommendation authoring; Pedestal Forum decree-addition drafting | None (recommendations are non-executing) |
| 4 | **fix-on-AUTHORITY_GRANTED** | Knight-class implementation of fixes via Cursor agent-mode subagent fan-out | **REQUIRED** — `AUTHORITY_GRANTED:<name>` token from Founder, keyed by canonical name + target innovation_id |

The 4-action pattern is **fail-safe by structural form**: three of four actions require no Founder gate (autonomous-class operation safe); only the 4th action (production-class change) requires explicit Founder Fire Code authority. This composes with the Founder Fire Code Universal Publication Gate canon ("NOTHING IS PUBLISHED without MY FIRE Code. So it's safe.") at the structural — not policy — level.

### 4.3 — Channel 4 → 5 → 6 Cascade

Action progression is implemented across three structurally-distinct Pheromone-substrate channels:

- **Channel 4 (directives)** — Queen-issued task assignments to Workers; Worker-issued analyze-class findings back to Queen; per-pair coordination directives. Carries phase 1 + phase 2 traffic.
- **Channel 5 (recommendations)** — Worker-authored recommendation packets; Pedestal Forum decree-addition drafts; cross-Worker recommendation cross-references. Carries phase 3 traffic.
- **Channel 6 (fix-execution-on-authority)** — fires **only** upon Founder dispatch of an `AUTHORITY_GRANTED:<name>` token. Each token is keyed by (i) canonical Old One name and (ii) target innovation_id. Channel 6 traffic is never emitted absent a matching authority token; the channel is structurally inert until activated. Carries phase 4 traffic.

The Channel 4 → 5 → 6 cascade is **monotonic-progression-only**: a pair cannot emit phase 4 traffic without having completed phase 3 + received the matching authority token. This constraint is enforced at the channel-listener level, not at the policy level — making channel-skip structurally impossible rather than merely prohibited.

### 4.4 — Cthulhu Arbitration (Queen / Coordinator)

The Queen — canonical placeholder name **Cthulhu**, recommended replacement **Cipher** — performs four (4) arbitration classes:

1. **File-conflict serialization** — when two or more Workers target overlapping files, Cthulhu serializes writes by Worker priority + first-arrival timestamp; losing Worker is restarted on a delta context.
2. **Dependency-ordering gating** — Cthulhu maintains the fleet's dependency DAG; a Worker whose target depends on an upstream innovation_id not yet landed is blocked at phase 1; un-blocked when upstream lands an Iron Tablet for the prerequisite.
3. **Cross-pair recommendation reconciliation** — when two Workers produce contradictory phase-3 recommendations on the same target, Cthulhu surfaces both to Pedestal Forum as Mordecai-Esther co-equal-authority decrees rather than picking a winner.
4. **Authority-token routing** — Cthulhu receives Founder-issued `AUTHORITY_GRANTED:<name>` tokens, validates the token's name + innovation_id binding, and routes to the matching Worker's Channel 6 listener.

Cthulhu is *not* a policy-overlord; it is a structural-arbitration layer. It cannot grant authority itself — only the Founder can — but it ensures authority-tokens are routed correctly and that Worker contention is resolved without race-class outcomes.

### 4.5 — Iron Tablet Writeback

Each completed phase produces an Iron Tablet entry keyed by composite `<name>+<innovation_id>`. The Iron Tablet is atomic-fused permanent: once written, it is append-only-immutable per the platform's substrate-as-immutable-backup canon. The entry records:

- Worker canonical name + canonical-replacement-name (when migration completes)
- Target innovation_id
- Phase outcome (analyze / evaluate / recommend / fix)
- Cost-instrumentation receipt across the 4-stream taxonomy
- Member-share-back-pay-eligibility flag (always true per Cost+20% structural-bylaw)
- Pheromone-event provenance hash chaining the entry to its Channel 4/5/6 source events

The Iron Tablet writeback is **per-pair-atomic**: a pair commits its tablet only when its phase has fully completed; partial writes are forbidden by the writeback contract.

**Claim 1.** A method for cooperative-class autonomous platform self-correction comprising: (a) instantiating an N-pair fleet of bidirectional orchestration cohorts wherein each cohort comprises a Bishop-tier orchestrator, a Knight-tier executor, a Shadow-tier autonomous-build agent, and a Subagent-tier fan-out layer; (b) assigning each cohort a specialization class drawn from the set { analyze, evaluate, recommend, fix }; (c) executing a monotonic 4-action loop across the fleet wherein the first three actions are autonomous-safe and the fourth action is gated by an explicit human-issued authority token keyed by cohort canonical name and target innovation identifier; (d) writing per-cohort phase outcomes back to an append-only-immutable substrate keyed by composite of cohort name and target innovation identifier.

**Claim 2.** The method of Claim 1, wherein the fleet comprises an 8-member core composed of one Queen / Coordinator cohort and seven Worker cohorts, the Queen cohort performing arbitration of cross-cohort file conflicts, dependency-ordering gates, contradictory-recommendation reconciliation via co-equal-authority append-only decree-composition, and authority-token routing.

**Claim 3.** The method of Claim 1, wherein action progression is enforced across three structurally-distinct event-substrate channels, the third channel being structurally inert until activated by receipt of an authority token matching the receiving cohort's canonical name, such that channel-skip is structurally impossible rather than policy-prohibited.

**Claim 4.** The method of Claim 1, further comprising a 4-stream cost-instrumentation taxonomy applied per cohort, wherein each cohort's cost cycle is independently member-share-back-pay-eligible at a Cost+20% structural margin, preserving cooperative-class anti-extraction structural form at fleet scale.

**Claim 5.** The method of Claim 1, wherein cohort-authored recommendations contradicting one another on a shared target innovation identifier are surfaced to a Pedestal Forum substrate as co-equal-authority append-only-immutable decree-additions in a Mordecai-Esther decree-composition pattern, rather than reconciled to a single victor recommendation.

**Claim 6.** A system implementing the method of Claim 1, comprising: a fleet-factory module operable to instantiate N parallel bidirectional orchestration cohorts; a per-cohort specialization-class binding independent of cohort canonical surface name and surviving atomic-rename operations across at least ten (10) substrate touchpoints; a Queen-class arbitration module operable to serialize cross-cohort file-write contention and to maintain a fleet dependency directed-acyclic-graph; a three-channel event substrate enforcing monotonic action progression; an authority-token validator binding each token to a cohort canonical name and a target innovation identifier; and a per-cohort Iron-Tablet writeback contract producing atomic-fused append-only-immutable phase outcome records.

**Claim 7.** The method of Claim 1, wherein the fleet operates concurrently on N distinct platform-sites of an existing cooperative-platform deployment, and wherein the architecture inverts a centralized-extractive continuous-integration pattern by operating instead as a cooperative-class member-share-back-pay-eligible self-correction architecture.

**Claim 8.** The method of Claim 1, wherein empirical fleet-scale throughput on an equivalent workload is at least five times (≥5×) the throughput of a singular-pair baseline cohort operating on the same workload, said multiplier being a structural property of N-pair concurrency under bounded Queen arbitration with serialized authority-gated production-change boundary.
