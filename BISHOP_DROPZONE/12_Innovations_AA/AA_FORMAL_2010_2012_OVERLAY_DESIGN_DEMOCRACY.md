## Innovation #2010 — Element-Level Design Overlay: "You Can Do Better" Visual Submission System

### Description

A system and method within a cooperative commerce platform wherein:

(a) When a member activates X-Ray Goggles (builder mode), every platform element with a `data-xray-id` attribute displays a cyan outline and a "You Can Do Better!" indicator — but ONLY in X-Ray mode. The indicator appears inline with the border of each asset that has Larks (improvement bounties) attached.

(b) Clicking "You Can Do Better" on a specific element toggles an OVERLAY directly on top of that element. The overlay is a canvas where the member can:
  - Write in their proposed replacement text/content
  - Upload an image or mockup that COVERS what's currently there
  - Sketch or annotate the existing element
  - Submit a code snippet (HTML/CSS) that replaces the element's rendering

(c) The overlay creates a **visual diff** — the current version is visible underneath, and the member's proposed replacement sits on top. Other members can toggle between "current" and "proposed" views.

(d) Submitted overlays become Larks — improvement proposals that earn Credits + Marks if accepted. The community votes on whether the proposed overlay is better than the current version.

(e) Multiple overlays can be submitted for the same element. Members vote between the current version and all proposed overlays. The winning version becomes the new default.

### The Flow

```
X-Ray Goggles ON
  → "You Can Do Better!" appears on elements with Larks
  → Click it → OVERLAY activates on that specific element
  → Write/upload YOUR version over what's there
  → Submit as Lark → earn Credits + Marks if accepted
  → Community votes: current version vs. your overlay
  → Winner becomes the new default
```

### Patent Relevance: HIGH

A cooperative commerce platform wherein members can propose visual replacements for specific UI elements through an overlay system activated in X-Ray inspection mode, with the proposed replacement rendered directly on top of the existing element creating a visual diff — and community voting determining which version becomes the platform default — represents a novel approach to member-governed interface design.

## Innovation #2011 — Community-Governed Visual Design: CSS Zen Garden for Cooperative Platforms

### Description

A system and method within a cooperative commerce platform wherein:

(a) Members can submit entire alternative page layouts and visual styles — complete CSS/theme overrides — that transform the platform's visual presentation while preserving the same underlying content and functionality. This follows the CSS Zen Garden model: same HTML, radically different CSS.

(b) Submitted themes are:
  - **Previewed** in a sandbox (the member sees the platform through their proposed theme)
  - **Published** to a gallery where other members can browse and preview them
  - **Voted on** by the community using the existing 6-level voting system
  - **Applied** as the member's personal view (any member can choose any approved theme for themselves)
  - **Promoted** to portal defaults through governance voting (the community decides the default look)

(c) This turns the platform's visual design into a **community-governed, living experiment**. The platform doesn't have ONE look — it has as many looks as the community creates, with governance determining the default.

(d) Theme creators earn:
  - **Marks** for each member who activates their theme (effort-differential)
  - **Credits** if their theme is voted as a portal default
  - **Reputation** in the Design Pipeline (Arena → Emporium → Crew Tables progression)

(e) Ghost Rules apply: non-members can preview themes for 24 hours. Members can permanently apply any approved theme. Theme creators must be members.

### Why This Is Academically Novel

CSS Zen Garden (est. 2003) proved the concept: same content, infinite presentations. But CSS Zen Garden was a design showcase, not a governance system. Liana Banyan's implementation adds:

1. **Democratic selection** — Members vote on which theme becomes the default, not a design committee
2. **Economic incentive** — Theme creators earn Marks and Credits, creating a design economy
3. **Cooperative ownership** — The community owns its visual identity collectively
4. **Portal differentiation** — Each of the 9 portals can have a different community-selected theme
5. **Living evolution** — Themes can be updated, forked, and evolved by the community over time

No existing platform — cooperative or otherwise — has implemented community-governed visual design through democratic voting with economic incentives for theme creators.

### The CSS Zen Garden Parallel

```
CSS ZEN GARDEN (2003)          LIANA BANYAN (2026)
────────────────────           ────────────────────
Same HTML content              Same platform content
Different CSS styles           Different member-created themes
Design showcase                Governance system
No voting                      Community voting
No incentives                  Marks + Credits for creators
Static gallery                 Living, evolving, governed
One site                       9 portals, each with its own default

```

### Patent Relevance: VERY HIGH — PAPER CANDIDATE

A cooperative commerce platform wherein members submit complete visual theme alternatives that transform the platform's presentation while preserving content and functionality, with community governance determining which theme becomes the default through democratic voting, and theme creators earning cooperative currency (Marks, Credits) based on adoption — creating a self-governing design ecosystem where the platform's visual identity is collectively owned and democratically evolved — represents a fundamentally novel application of cooperative governance to digital interface design.

## Innovation #2012 — Design Democracy Voting: Threshold Governance for Visual Identity

### Description

A system and method within a cooperative commerce platform wherein:

(a) Visual design changes — from element-level overlays (#2010) to full page themes (#2011) — are governed through the platform's existing voting infrastructure:

  - **Element overlays** use Document Voting (reputation-only, no credits spent):
    - +10 net votes: overlay replaces the current element for all members
    - -5 net votes: overlay hidden
    - Creator of accepted overlay earns Marks
  
  - **Page themes** use 6 Production Level voting (credit-weighted):
    - Level 1 (Pre-Mint, 0-99 credits): Theme is in gallery, early voters get 5x multiplier
    - Level 2 (Minted, 100-999): Theme available for personal use
    - Level 3 (Production, 1,000-4,999): Theme eligible for portal default nomination
    - Level 4 (Distribution, 5,000-9,999): Theme becomes a portal default option
    - Level 5 (Established, 10,000+): Theme is the community-selected portal default

(b) **Portal-level governance**: Each of the 9 portals (.com, .biz, .org, .net, the2ndsecond, hexisle, hexislo, dss, upekrithen) votes on its own default theme independently. The Marketplace might choose a warm, inviting theme while the Captain/Business portal chooses a professional, data-dense theme.

(c) **Season system**: Themes can be nominated for seasonal rotation (spring, summer, fall, winter) or event-based activation (Opening Gambit launch, holiday seasons, milestone celebrations). The community votes on seasonal themes in advance.

(d) **Fork and evolve**: Any member can fork an existing theme, modify it, and submit the fork as a new theme. This creates a design evolution tree where the best elements of multiple themes get combined through community experimentation.

### Patent Relevance: HIGH

A cooperative commerce platform wherein visual design governance follows a tiered democratic process — with element-level changes governed by reputation voting and page-level themes governed by credit-weighted production levels — across multiple portal surfaces that each maintain independent visual identity through separate governance tracks — represents a novel multi-scale democratic design governance system.
